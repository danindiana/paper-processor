## Motivation

During review of `paper_processor.py` it became clear that the pipeline never
passes image data to any model:

- `fitz.get_text("text")` discards all embedded PDF images.
- `Backend.call(prompt: str, ...)` accepts only a text prompt.
- No multimodal fields exist in either the Ollama or OpenClaw request payloads.

The `xl_quality` tier was set to `gemma4:31b-it-q4_K_M` — a multimodal model
holding ~18 GB VRAM for capabilities that are never used.

## Change

`MODEL_TIERS["xl_quality"]` → `gpt-oss:20b`

- Text-only model, no vision overhead.
- Estimated VRAM: ~12 GB (saves ~6 GB on the dual-GPU pool).
- Routing threshold unchanged: papers >18 pages still route to `xl_quality`.

## VRAM expectations after swap

| Model | VRAM |
|-------|------|
| `gpt-oss:20b` (xl_quality) | ~12 GB |
| `qwen3-coder:30b` (xl_code) | ~17 GB |
| Both loaded simultaneously | ~29 GB — exceeds 26.5 GB combined; normal eviction applies |

Single-model operation is the default (`OLLAMA_MAX_LOADED_MODELS=1`), so
eviction between `xl_quality` and `xl_code` continues as before.
