## Incident note: dangerous interim state

Between `chsh` succeeding and `apt-get install` actually running, the
machine spent several minutes with its login shell pointing at a
non-existent binary (`/usr/bin/zsh` did not exist; `/etc/shells` did not
list zsh). Already-running bash sessions kept working because the kernel
holds the running binary, but a fresh login (TTY, ssh, new graphical
terminal in some setups) would have failed.

Cause: a non-TTY harness can't pass passwords to `sudo`, so two attempts
to run `apt-get install` from this session silently failed
(`sudo: A terminal is required to authenticate`). The `chsh` step had
been issued through the same path on a prior turn and was reported as
complete by the user without verification.

For future sessions on `morlok`:

- **Order of operations**: install the new shell binary **before** running
  `chsh`. Even if the change is reported as done, verify
  `which <shell>` and the file actually exists before changing the login
  shell.
- **Sudo from Claude on morlok**: the `!` prefix uses the same non-TTY
  bash and cannot answer a password prompt. Sudo work must be run by the
  user in a real terminal. Don't trust "I ran it" — verify with
  `dpkg -l <pkg>` or `which <bin>`.

## How to use

For the **current** bash session, the change does not take effect — the
shell was set at fork time. Open a new terminal (or run `exec zsh -l`) to
land in zsh. From there:

```zsh
cd /path/to/some/project        # must contain .venv/ or venv/
v                                # activates the venv
deactivate                       # leaves it
```

Adding more PATH entries or aliases later: edit `~/.zshrc`, then either
open a new terminal or `source ~/.zshrc`.
