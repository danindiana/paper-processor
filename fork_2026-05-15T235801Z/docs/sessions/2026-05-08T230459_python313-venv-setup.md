### 2. Added a `v` activate-helper to `~/.bashrc`

Appended after the existing `PATH` block:

```bash
# Activate the Python venv in the current directory (.venv preferred, then venv).
# Usage: cd into a project, then type `v`.  `deactivate` to leave.
v() {
    if [ -n "$VIRTUAL_ENV" ]; then
        echo "Already in venv: $VIRTUAL_ENV  (run 'deactivate' first)"
        return 0
    fi
    for d in .venv venv; do
        if [ -f "./$d/bin/activate" ]; then
            # shellcheck disable=SC1090
            . "./$d/bin/activate"
            echo "activated: $VIRTUAL_ENV  ($(python --version 2>&1))"
            return 0
        fi
    done
    echo "v: no .venv or venv found in $PWD" >&2
    return 1
}
```

Behaviour:

- Reusable across **any** project — looks for `.venv/bin/activate` first, then
  `venv/bin/activate` in the current directory.
- Idempotent — refuses to re-activate if `$VIRTUAL_ENV` is already set; tells
  the user to `deactivate` first.
- Fails loudly with a non-zero exit code and a stderr message when no venv is
  found, so it's safe to use in scripts.
- Only the parent shell's environment is modified (we use `.` / `source`,
  not a subshell).

## How to use

For an already-open shell, reload `~/.bashrc` once:

```bash
source ~/.bashrc
```

After that, in any new terminal:

```bash
cd /path/to/some/project        # must contain .venv/ or venv/
v                                # activates it
python --version                 # -> Python 3.13.x (for this project)
deactivate                       # leave the venv
```

To create a venv for a brand-new project so `v` works there too:

```bash
cd /path/to/new/project
python3.13 -m venv .venv
v
```

## Notes / caveats

- `bash -c '...'` (non-interactive) will **not** see `v` because `~/.bashrc`
  early-returns for non-interactive shells. That's expected — `v` is for
  human terminal sessions. Use `bash -i -c` if you really need to test it
  from a one-liner.
- The function does **not** auto-activate on `cd`. That was an explicit
  choice: auto-activation surprises users when `cd`-ing into nested project
  trees and can clobber an already-active venv. Pressing `v` is one keystroke
  and is unambiguous.
- If multiple venvs are ever needed in one project, keep the canonical one
  at `.venv` and activate the others manually with
  `source ./other-venv/bin/activate`.
- The CLAUDE.md at the drive root describes the `worlock` host (user `jeb`),
  but this session ran on the local machine as user `morlok` against the
  WORLOCK drive mounted at `/run/media/morlok/WORLOCK/`. The bashrc edit
  therefore went to `/home/morlok/.bashrc`, not to anything on the WORLOCK
  drive.
